# AI-Powered Speech Analytics Engine (Система речевой аналитики звонков)

Асинхронный микросервисный конвейер для автоматического контроля качества телефонных звонков отдела продаж. Система локально распознает аудиозаписи в текст и с помощью LLM извлекает ключевые бизнес-метрики: оценивает вежливость менеджера, фиксирует возражения клиентов и формирует краткое саммари разговора.

---

## 🏗️ Архитектура и движение данных

Проект спроектирован по принципу **слоёной архитектуры (Layered Architecture)**, что обеспечивает слабую связанность компонентов и простоту поддержки:
[Клиент / CRM] ──► FastAPI (Слой API / Валидация через Schemas)
│
▼ (Задача ставится в очередь)
Redis (Брокер сообщений)
│
▼ (Воркер забирает задачу)
Celery Worker (Фоновые задачи)
├──► Faster-Whisper (Локальный движок STT)
└──► YandexGPT API (Аналитический сервис ИИ)
│
▼ (Результат сохраняется)
PostgreSQL (Слой данных / Models)

1. **FastAPI (Слой API):** Принимает HTTP-запросы, валидирует входные данные через Pydantic-схемы и мгновенно возвращает клиенту статус `202 Accepted` с уникальным ID задачи, не дожидаясь окончания обработки.
2. **Redis и Celery (Асинхронная обработка):** Исключают блокировку основного потока сервера (CPU-blocking). Тяжелые задачи транскрибации аудио и сетевые запросы к ИИ выносятся в изолированные фоновые воркеры.
3. **Сервисный слой (`services/`):** Содержит чистую бизнес-логику (`WhisperEngine` и `LLMAnalyzer`), полностью независимую от HTTP-слоя.
4. **PostgreSQL:** Хранит финальные результаты транскрипции и структурированный JSON-анализ звонков.

---

## 🛠️ Технологический стек

* **Бэкенд:** Python, FastAPI, Celery, Redis, PostgreSQL, SQLAlchemy, Pydantic
* **AI/ML (Речевая аналитика):** Faster-Whisper (с квантованием `small-int8` для оптимизации потребления памяти на CPU)
* **Интеграция с LLM:** YandexGPT API (с получением строго структурированного JSON-ответа)
* **Инфраструктура:** Docker, Docker Compose

---

## ⚙️ Отказоустойчивость и особенности инфраструктуры

* **Сетевая изоляция:** Архитектура полностью контейнеризирована. Сервисы изолированы внутри приватной сети Docker и общаются между собой по внутренним именам хостов (`host="db"`, `host="redis"`), что исключает конфликты портов на сервере.
* **Fault Tolerance (Фоллбэк для ИИ):** В архитектуру аналитического сервиса заложена строгая обработка исключений `try/except`. При деградации внешнего API, ошибках авторизации (например, 403 Forbidden) или сетевых сбоях со стороны провайдера ИИ, система автоматически подставляет валидационный шаблон-заглушку. Это гарантирует успешное завершение Celery-задачи без падения воркера и нарушения целостности базы данных.

## 🚀 Инструкция по локальному запуску

Проект полностью контейнеризирован, поэтому для запуска вам понадобятся только установленные **Docker** и **Docker Compose**.

### 1. Клонирование репозитория
```bash
git clone [https://github.com/holmirzaev/monolith_analytics.git](https://github.com/holmirzaev/monolith_analytics.git)
cd monolith_analytics
```

### 2. Настройка переменных окружения
Создайте текстовый файл с именем `.env` в корневой директории проекта и укажите ваши доступы:
```env
POSTGRES_USER=your_user
POSTGRES_PASSWORD=your_password
POSTGRES_DB=analytics_db
REDIS_URL=redis://redis:6379/0
YANDEX_FOLDER_ID=your_folder_id
YANDEX_API_KEY=your_api_key
```

### 3. Запуск инфраструктуры
Выполните команду для автоматической сборки и старта всех контейнеров:
```bash
docker compose up --build
```

После завершения сборки:
Интерактивная документация (Swagger API) доступна в [Swagger UI](http://localhost:8000/docs).
Сервис Celery и база данных PostgreSQL: автоматически развернутся в фоне и начнут обработку задач.
